Phase 1

Foundation & Discovery

Initial Challenge

üéØ The Challenge

Stanford had 50+ Page Builder components that needed to support inline editing in Squiz DXP's Visual Page Builder. The core challenge was converting components that used shared global utilities (cards, buttons, etc.) which needed different data-sq-field values depending on the component context.

The Core Problem

Shared Template
<h3 data-se="title">{{title}}</h3>
<p data-se="description">{{description}}</p>

One template, multiple contexts

‚Üí
Component A Needs
data-sq-field="heroCards[0].title"
data-sq-field="heroCards[0].description"

Specific field paths

‚Üí
Component B Needs
data-sq-field="content.featuredTitle"
data-sq-field="content.featuredDesc"

Different field paths

üîç Discovery Process

1

Problem Identification

Shared partials needed dynamic field mapping without breaking existing functionality

2

Solution Research

Discovered the processSquizEdit utility approach for mapping generic attributes

3

Methodology Development

Established "Minimal Changes Only" principle to preserve front-end behavior

üí° Key Insight

"Shared global utilities need different data-sq-field values depending on which component uses them, but they share the same template structure."

This insight led to the development of the processSquizEdit mapping approach.

Phase 2

Component Conversion Methodology

Systematic Approach

üîß The ProcessSquizEdit Approach

We developed a systematic 5-step methodology that became the foundation for all component conversions:

1

Import the Utility Function

‚ñº 
import { processSquizEdit } from '../../global/js/utils/isEditor';

Only import processSquizEdit - no other utilities needed!

2

Detect Edit Mode

‚ñº 
const squizEdit = info?.ctx?.editor || false;

Simple boolean check - true in edit mode, false in front-end mode.

3

Make Variables Mutable

‚ñº 
// CHANGE: const to let for mutability
let { title, content } = args || {};

Variables need to be mutable so we can add defaults in edit mode.

4

Configure Edit Targets

‚ñº 
let squizEditTargets = null;
if (squizEdit) {
    title = title || 'Add your title';
    content = content || 'Add content';
    
    squizEditTargets = {
        "title": { "field": "title" },
        "content": { "field": "content" }
    };
}

Only configure in edit mode. Always provide defaults for empty fields.

5

Early Return Pattern

‚ñº 
const componentData = { title, content };

if (!squizEdit) return template(componentData);

return processSquizEdit(template(componentData), squizEditTargets);

Clean separation: front-end gets original template, edit mode gets processed version.

üìã Three Configuration Formats Identified

Format 1: Simple

One-to-one attribute mapping for unique elements

"title": { "field": "title" }

Format 2: Array

Automatic index generation for homogeneous collections

"button": {
    "field": "buttons",
    "array": true,
    "property": "buttonText"
}

Format 3: Custom

Multiple elements with same data-se but different field paths

"button": [
    { "field": "colTwo.buttonText" },
    { "field": "colThree.buttonText" }
]

üéØ Matrix Asset ID Standardization

Objective: Update all squizEdit image default matrix-asset values to use ID 130444

Target Format: matrix-asset://matrixIdentifier/130444

Before

Multiple IDs in use:

1 2 3 5 6 7 8 9 10 11 14
‚Üí

After
